写完前面几篇 Agent 笔记之后，我开始对一个具体的设计模式越来越好奇：**当一个 Agent 不够用的时候，怎么办？**

如果你用过 Claude Code 一段时间，大概会注意到一件事 —— 它在干一些复杂活的时候，并不是闷头自己跑，而是会「派出去」几个子任务，让别的 Agent 去做，自己再汇总结果。这种「一个主 Agent 带着一堆子 Agent 干活」的模式，社区里通常叫做 **manage agent**（管理式智能体），也有人叫 orchestrator pattern（编排者模式）、manager-worker pattern。

这一篇就是我对这个模式的理解笔记：**它到底是什么，解决了什么问题，又有哪些隐藏的成本。**

## 一句话：把一个 Agent，拆成一个团队

回顾一下我之前写过的 Agent 心跳：

```
目标 → [ 思考 → 行动 → 观察 ] → ...... → 完成
```

这套循环对**一个**目标、**一条**主线任务很管用。但现实里的任务往往不是一条线 —— 它更像是一张网：要同时查几个地方、要并行做几件事、要在不同领域之间来回切换。

Manage agent 模式的核心想法很朴素：

> **让一个「主 Agent」专注于决策和编排，把具体的脏活分给若干「子 Agent」去做。**

主 Agent 像项目经理，负责拆解任务、分工、收口；子 Agent 像专项小组，各自带着工具、上下文和目标，干完自己那一份，把结果交回来。

用更工程化的话说：这是把**单线程的 Agent 循环**，变成了一个**有层级的多 Agent 系统**。

## 它要解决的三个真问题

我一开始也疑惑：一个 Agent 不就够了吗？多搞几个不就是多花钱吗？

直到自己跑过几次长任务，才慢慢看明白这个模式在解什么问题。

### 一、上下文窗口是有限的

哪怕模型的上下文窗口被卷到了一百万 token，它仍然是**有限**的；而且更关键的是，**上下文里塞的东西越多，模型的注意力就越容易被稀释。**

举个我亲身踩过的例子。让 Claude Code 在一个大仓库里找「某个功能是怎么实现的」，如果它直接读文件，几十个文件读下来，主对话里就堆满了原始代码 —— 接下来再让它做点别的，它会变迟钝、会忘记最初的目标、会被某段无关的代码带偏。

manage agent 模式的解法是：**把搜索这件脏活交给一个子 Agent。** 子 Agent 自己去翻几十个文件，最后只把一段简短的总结交回来。主 Agent 的上下文里，只多了「这段总结」，而不是「几十个文件的原文」。

> 子 Agent 像一层「上下文防火墙」，把原始信息挡在外面，只让结论穿过来。

这一条听起来不起眼，但在长任务里几乎是决定性的 —— **谁能更省着用上下文，谁就能跑得更久、想得更清楚。**

### 二、有些任务天然可以并行

主 Agent 一次只能想一件事，但任务本身常常是可以并行的：同时查三个文件、同时跑两个搜索、同时让两个评审角度看同一份代码。

如果只有一个 Agent，这些事只能排队做。换成 manage agent 模式，主 Agent 可以**一次性派出几个子 Agent**，让它们并发去跑，自己等结果。任务总耗时被压缩到「最慢的那个子任务」，而不是「所有子任务之和」。

这是一种朴素但有效的加速：**用模型的并行调用，换人的等待时间。**

### 三、不同任务需要不同的「专长」

通用模型很强，但在具体任务上，**有针对性的提示词 + 有限的工具集**往往效果更好。

Claude Code 里能看到一组明显有「分工」的子 Agent，大致是这种思路：

| 子 Agent | 它的专长 |
| --- | --- |
| Explore | 只读、快速定位代码：找文件、grep 关键词、回答「X 在哪定义」 |
| Plan | 软件架构师视角，做实现方案规划，不写代码 |
| general-purpose | 通用研究型，能搜索、能多步推理，适合开放性问题 |

每一个子 Agent 都有自己的「人设」、自己的可用工具、自己的工作边界。主 Agent 不需要什么都自己会 —— 它只需要**知道在什么时候叫谁**。

这有点像现实里的团队：让前端工程师去查前端 bug，让 DBA 去看慢查询，比让一个全栈一个人扛要快、也更靠谱。

## 它怎么跑：一个简化的循环

把 manage agent 的运转拆开看，大致是这样：

```
            ┌─────────────────────────────┐
            │       主 Agent（管理者）       │
            │  - 拆解任务                   │
            │  - 决定派谁去                 │
            │  - 汇总结果，决定下一步        │
            └──────────────┬──────────────┘
                           │ 任务下发（含目标 + 边界）
            ┌──────────────┼──────────────┐
            ▼              ▼              ▼
        ┌────────┐   ┌────────┐    ┌────────┐
        │ 子 Agent │   │ 子 Agent │   │ 子 Agent │
        │   #1    │   │   #2    │   │   #3    │
        └────┬────┘   └────┬────┘   └────┬────┘
             │             │              │
             └─────────────┴──────────────┘
                       结果回流（精简后的结论）
```

几个关键点：

1. **主 Agent 不下场**。它本身不去读文件、不去 grep —— 那是子 Agent 该干的。它的角色是**调度和判断**。
2. **任务下发要自包含**。子 Agent 没有主对话的上下文，所以主 Agent 必须把背景、目标、约束一次性讲清楚 —— 就像你给一个新同事派活，不能只说「你懂的，去做一下」。
3. **结果是经过浓缩的**。子 Agent 返回的，不是它读到的所有原始材料，而是一份**结论性的摘要**。这是 manage agent 模式省上下文的关键。
4. **并行优于串行**。如果几个子任务之间没有依赖，主 Agent 应该一次性把它们都派出去，而不是一个个等。

## 我对它的几个直觉

跑了一阵之后，我对这个模式有了几个比较稳定的判断。

**它不是「越多越好」。** 子 Agent 不是免费的：每派出去一个，就是一次额外的模型调用、一段额外的提示词、一份额外的延迟。简单任务硬塞 manage agent，结果往往是**把简单的事做慢做贵**。

**它是「上下文经济学」。** 我现在更愿意这样理解 manage agent：它本质上是在做**上下文预算管理**。主 Agent 的上下文最贵 —— 因为它要做最重要的决策；所以一切能挤进子 Agent 上下文里的脏活，都应该挤进去。

**它对「任务描述」的要求陡然提高。** 一旦你开始派子 Agent，写好那段「任务描述」就成了核心技能。模糊的任务 → 模糊的结果 → 主 Agent 拿到的是噪声而不是信号。写好子任务提示词，几乎和写好系统提示词一样重要。

**它把人的角色推得更高了。** 过去你盯一个 Agent 干一件事；现在你盯一个 Agent 在指挥一群 Agent 干一摊事。你看不见每个子 Agent 内部的步骤，只能看到它们的结论。这意味着 —— **你必须更早、更清晰地定义「什么样的结果算合格」**，否则错误会被一层一层包装得很漂亮地交上来。

## 几个真实的坑

我自己踩过、或者看到别人踩过的几个：

- **过度委派。** 一个简单的「打开这个文件看看」也派一个子 Agent，主 Agent 反而被来回的派发和等待拖慢。判断标准很简单：**这件事我直接做，比派出去再等回来更慢吗？** 如果不是，就别派。
- **子 Agent 的「自信幻觉」**。子 Agent 返回的是一段总结，看起来很笃定。但你要记得：它**只看到了它自己读到的东西**。它可能漏了一个目录、可能没注意到一个分支。主 Agent（以及你）不能把子 Agent 的总结当成绝对真相。
- **任务边界不清。** 派两个子 Agent 做相似但又不完全一样的事，结果两份报告大半重复，关键差异被淹没。**好的任务划分应当让子任务的输出彼此互补，而不是彼此重复。**
- **失去可观测性。** 子 Agent 内部的几十步推理，主 Agent 看不到，你也看不到。任务出问题时，调试链路被拉长了。所以越是关键的步骤，越要让它显式地把「过程」也带回来一点，而不是只给结论。

## 它和我之前学的东西怎么对上

把这几篇笔记串起来看，其实有一条很清晰的线：

- 第一篇讲 **Agent = 模型 + 循环 + 工具** —— 这是单 Agent 的内核；
- 第二篇讲 **Hermes 用持久记忆把单 Agent 拉长** —— 解决「时间维度」的扩展；
- 这一篇讲 **manage agent 把单 Agent 拉宽** —— 解决「空间维度」的扩展。

一个 Agent 想要做更复杂的事，要么活得更久（记忆），要么调动更多帮手（管理）。两件事都是在跟「上下文有限」这个根本约束对抗。

而 manage agent 的可贵之处在于：**它没有让单个 Agent 变得更复杂，而是让 Agent 之间的协作变得更清晰。** 复杂度被分摊到了「分工」上，而不是堆在某一个 Agent 的脑子里。

## 小结

照例三句话留给自己：

- Manage agent ≈ **一个主 Agent + 若干各有专长的子 Agent**，主 Agent 负责调度，子 Agent 负责脏活。
- 它真正解决的是**上下文经济学**：把原始信息挡在子 Agent 里，让主 Agent 始终保持清醒。
- 它不是银弹 —— 任务一旦变成「派人 + 等结果」，**写清任务、定义结果**就成了比写代码还重要的事。
