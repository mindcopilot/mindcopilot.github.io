这是我学习 AI Agent 的一份笔记。

过去一段时间，「Agent」（智能体）几乎成了 AI 圈最热的词。但热词的麻烦在于：每个人都在说，却很少有人把它说清楚。我自己也被绕晕过 —— 直到真正动手用过几个 Agent，才慢慢分清哪些是本质，哪些只是包装。

这篇文章就是我整理给自己的答案：**AI Agent 到底是什么，它怎么运转，以及一个普通学习者该从哪里上手。**

## 一句话：Agent 是「会做事」的模型

最朴素的区分是这样的：

- 一个普通的对话模型，你问它问题，它给你答案 —— 它**会聊天**。
- 一个 Agent，你给它一个目标，它会自己拆解步骤、调用工具、执行、检查结果，直到把事情做完 —— 它**会做事**。

> 聊天模型输出的是「文字」；Agent 输出的是「结果」。

举个例子。你问聊天模型「我这个月在咖啡上花了多少钱」，它只能说「我看不到你的账单」。而一个 Agent 可以：读取你的账单文件 → 筛选咖啡相关条目 → 求和 → 把数字告诉你。差别不在于模型更聪明，而在于它**被允许行动**。

## 从 Prompt 到 Agent：三级跳

我把自己理解 Agent 的过程，拆成了三个台阶：

**第一级 · 单轮问答。** 一问一答，模型只靠它训练时学到的知识。够用，但封闭 —— 它不知道今天的日期，也碰不到你的数据。

**第二级 · Prompt 工程。** 你学会了把上下文、例子、要求一起喂给模型，让回答更准。这一步很重要，但本质上还是「一次性」的：模型说完就结束了。

**第三级 · Agent。** 模型被放进一个**循环**里，并且拿到了**工具**。它可以说「我需要查一下」，于是真的去查；查到结果后再决定下一步。它不再是回答一个问题，而是**推进一个任务**。

关键的跃迁发生在第二级到第三级之间，而这个跃迁靠的是两样东西：**循环**和**工具**。

## Agent 的核心：一个循环

把花哨的框架名词都剥掉，几乎所有 Agent 跑的都是同一个循环：

```
目标 → [ 思考 → 行动 → 观察 ] → ...... → 完成
```

具体来说：

1. **思考（Reason）**：模型根据当前目标和已知信息，决定下一步该做什么。
2. **行动（Act）**：它调用一个工具 —— 搜索、读文件、运行代码、调用 API。
3. **观察（Observe）**：工具返回结果，这个结果被加回上下文。
4. **重复**：带着新信息，再想一次、再做一次，直到目标达成。

这个「思考—行动—观察」交替进行的模式，研究上叫 **ReAct**（Reasoning + Acting）。名字可以忘掉，但这个循环值得记住 —— 它就是 Agent 的「心跳」。

我第一次真正「看懂」Agent，是在用编程类 Agent 的时候：它读一段报错 → 推测原因 → 改一行代码 → 重新运行 → 看新的报错 → 再改。那一刻我意识到，**它做的事和我自己 debug 时做的事，结构上一模一样。**

## 几个绕不开的概念

上手过程中，有四个概念反复出现，值得单独拎出来：

| 概念 | 它解决什么 |
| --- | --- |
| 工具 / 函数调用 | 让模型能「碰到」真实世界：查资料、读写文件、调 API |
| 记忆 | 上下文窗口是「短期记忆」；要记更久，得靠外部存储与检索 |
| 规划 | 把一个大目标拆成可执行的小步骤 |
| 多智能体 | 用多个分工的 Agent 协作，一个「调度」、几个「干活」 |

其中我觉得最容易被低估的是**记忆**。模型的上下文窗口再大也是有限的，任务一长，早期的信息就会被挤出去。所以「Agent 怎么记住该记的、忘掉该忘的」，本身就是一门手艺。

另外值得一提的是 **MCP（Model Context Protocol）** —— 一个让 AI 模型以统一方式连接工具和数据源的开放标准。在它出现之前，每接一个工具都要写一套适配；有了统一协议，工具生态才更容易长起来。

## 我是怎么上手的

如果让我给「想学 Agent 的自己」提建议，大概是这几条：

1. **先用，再拆。** 不要一上来啃论文。先去用一个成熟的 Agent 产品（比如编程类 Agent），亲眼看它怎么一步步推进任务。直觉建立起来后，概念会自己对上号。
2. **从单工具开始。** 自己动手时，别一上来就做多智能体。先让一个模型学会用**一个**工具，把「思考—行动—观察」这个最小循环跑通。
3. **盯着它的「思考过程」看。** Agent 最有价值的学习材料，是它每一步的推理日志。它在哪里想错了？为什么绕了远路？这些比成功案例更能教会你。
4. **给它设检查点。** 不要让 Agent 一口气跑完十步。在关键节点让它停下来、由你确认 —— 尤其是涉及「不可逆」操作的时候。

## 一点清醒：Agent 不是魔法

学得越多，我反而越克制。几个真实的坑：

- **错误会滚雪球。** 循环越长，早期一个小误判就可能被后面每一步放大。短任务往往比长任务更可靠。
- **它会一本正经地做错。** Agent 可能调用一个并不存在的工具，或对结果过度自信。它的「自信」和「正确」是两回事。
- **成本不只是钱。** 每多一轮循环，就多一次模型调用。一个设计糟糕的 Agent，能用很高的代价把简单的事做复杂。
- **评估很难。** 聊天回答对不对，一眼能看出来；一个跑了二十步的 Agent 从哪一步开始跑偏，要查清楚并不容易。

所以我现在的态度是：**把 Agent 当成一个能力很强、但需要被引导和复核的伙伴。** 它负责把事情推进得更快，而「目标定得对不对、结果可不可信」，仍然是我的责任。

## 小结

如果只让我留下三句话：

- Agent = **模型 + 循环 + 工具**，本质是让「会聊天」的模型变得「会做事」。
- 它的心跳，是**思考 → 行动 → 观察**的不断重复。
- 它很强，但不是魔法 —— **方向盘还得握在自己手里。**
