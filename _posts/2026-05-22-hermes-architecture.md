我最近想搞清楚一个问题：开源社区里口碑很好的 **Hermes** 系列大模型，它的「架构」到底长什么样？

带着「找一张神经网络结构图」的预期去查资料，结果第一个发现就把我的预期打翻了：

> Hermes 并不是一种新的模型架构。

而这，反而成了这篇笔记最有意思的起点。

## 先厘清：Hermes 不是一个新网络结构

Hermes 是 Nous Research（以 Teknium 等人为代表的开源 AI 团队）推出的一个模型系列。但严格说，它**不是从零设计、从零训练**的模型。

它是在别人的**基座模型（base model）**上做**微调（fine-tune）**得到的：

- 早期的 Nous-Hermes 基于 Meta 的 LLaMA；
- Nous Hermes 2 用过 Mistral、Mixtral 等基座；
- 到了 Hermes 3，基座换成了 Llama 3.1，覆盖 8B、70B、405B 几个规模。

也就是说，Hermes 的 Transformer 主体 —— 注意力机制、层数、参数规模 —— 基本是**从基座模型继承来的**。你要真想看「网络结构图」，那张图其实是 Llama 的。

那问题就来了：既然网络结构是借来的，为什么大家还把 Hermes 当成一个有辨识度的模型？它的「架构原理」又该从哪儿看？

我的答案是：**Hermes 的设计，几乎全部发生在「训练之后」。** 它真正的架构，是一套**后训练（post-training）的方法体系**。下面拆成四层来看。

## 第一层：数据即架构

如果只能用一句话概括 Hermes 的理念，我会选这句：**模型的行为，主要由微调数据决定。**

Hermes 这条线最核心的资产，其实是它的**数据集**（围绕 OpenHermes 数据集发展而来）。它大致有三个特点：

- **以合成数据为主**：大量样本由更强的模型生成，再经过筛选；
- **规模可观**：到 OpenHermes 2.5 这一代，指令样本已达到百万量级；
- **来源广、清洗重**：把许多公开数据源汇总、去重，过滤掉低质量和「机器味」太重的内容。

换句话说，与其说团队在「设计模型」，不如说他们在**设计模型要读的教材**。这就是我理解的「数据即架构」—— 在基座能力给定的前提下，数据集的构成，决定了模型最终是什么脾气、擅长什么。

## 第二层：训练配方

有了数据，还需要一套把数据「喂」进模型的流程。Hermes 用的是当下开源微调里相当主流的两步：

1. **SFT（监督微调）**：用「指令 + 理想回答」的成对数据，教模型按期望的方式应答。这一步决定了模型的基本能力和风格。
2. **DPO（直接偏好优化）**：再用「更好 / 更差」的成对回答，让模型在偏好上对齐。相比传统 RLHF，DPO 不需要单独训练一个奖励模型，流程更简单、也更稳定 —— 这正是近两年开源社区普遍转向它的原因。

值得一提的是，Hermes 3 是对 Llama 3.1 的**全参数微调**，而不是只更新一小部分参数的轻量微调（如 LoRA）。全参数微调更「彻底」，代价是算力开销大得多 —— 这本身也是一种架构取舍。

## 第三层：对话与工具的「接口设计」

这一层我觉得最配得上「架构」二字，因为它真的是在设计**协议和接口**。

**统一的对话格式。** Hermes 系列坚持使用 **ChatML** 格式 —— 用特殊标记把对话切成 `system` / `user` / `assistant` 几种角色，结构清晰、多轮一致。格式统一看似是小事，但它让「系统提示词」有了稳定的落脚点，也为后面的工具调用打好了地基。

**结构化的工具调用。** 从 Hermes 2 Pro 开始，模型内建了对**函数调用 / 工具使用**的支持，并约定了一套清晰的格式：

```
系统提示里用 <tools> ...... </tools> 声明可用工具
模型需要调用时，输出 <tool_call> ...... </tool_call>（内含 JSON）
工具执行结果再用 <tool_response> ...... </tool_response> 回填
```

如果你读过我上一篇讲 AI Agent 的笔记会发现：这套「声明工具 → 发起调用 → 回填结果」的格式，正是把一个模型变成 Agent 的关键接口。Hermes 把它**固化进了训练**，让模型「天生」就懂这套协议，而不是靠临时的提示词去硬凑。

此外，它也强化了**结构化输出**（比如严格的 JSON）能力 —— 同样是为了让模型更好地嵌进程序里。

## 第四层：可控性哲学

最后一层不是技术，而是**价值取向**，但它实实在在地塑造了 Hermes。

Hermes 一个被反复强调的设计目标是**可控、可被引导**：模型应当尽量忠实地服从**系统提示词**，而不是被训练方预先塞进一堆固执的「个性」与拒绝行为。

用我自己的话说：**Hermes 想做一件趁手的工具，而不是一个有主见的对话者。** 方向，由使用它的人来定。

这个取向有它的好处 —— 对开发者友好，适合搭各种 Agent 和应用；也有它的代价 —— 更少的内置约束，意味着更多的责任落到使用者身上。这恰好呼应了我一直在想的那件事：**当工具越来越强、越来越「听话」，怎么用好它，就越来越是人的责任。**

## 这对我意味着什么

把四层拼起来，我对「Hermes 架构原理」的理解变成了这样：

> Hermes ≈ 借来的基座（Llama 等） + 精心设计的数据 + 成熟的训练配方（SFT + DPO） + 清晰的对话与工具接口（ChatML / 函数调用） + 可控性优先的哲学。

对我这样一个学习者，最大的收获是一次**视角的转变**：

- 过去我以为「模型的架构」就是那张神经网络结构图；
- 现在我更愿意相信：在开源大模型的世界里，**真正拉开差距的「架构」，越来越多地发生在训练之后** —— 藏在数据、配方、接口和理念里。

基座模型像毛坯房，人人拿到的都是同一套；而 Hermes 做的，是一整套**精装修方案**。

## 小结

三句话留给自己：

- Hermes 不是新的网络结构，而是基座模型之上的**后训练方法体系**。
- 它的四层「架构」：数据、训练配方、接口设计、可控性哲学。
- 在开源模型时代，**会读架构图固然好，但更要会读「训练之后」发生的事。**
